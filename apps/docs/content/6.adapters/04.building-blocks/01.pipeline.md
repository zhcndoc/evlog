在生产环境中，为每条日志事件发送一次 HTTP 请求是很浪费的。drain pipeline 会缓存事件并批量发送，在短暂故障时重试，并在缓冲区溢出时丢弃最旧的事件。

将我的 evlog drain 包装到共享 pipeline 中（批处理 + 重试 + 缓冲区溢出保护）。

- 识别我的框架，并遵循其 evlog 绑定模式
- 从 'evlog/pipeline' 导入 createDrainPipeline，并导入适配器（例如 evlog/axiom）
- 包装适配器：const drain = createDrainPipeline<DrainContext>()(createAxiomDrain())
- 配置 batch（{ size, intervalMs }）、retry（{ maxAttempts, backoff }）以及 bufferSize 选项
- 使用 fan-out 同时向多个 drain 发送：pipeline(...adapters)
- 在框架关闭 / SIGTERM 时挂钩 drain.flush()，确保缓冲事件不会在关闭时丢失

文档：https://www.evlog.dev/adapters/building-blocks/pipeline
适配器：https://www.evlog.dev/adapters/overview

::

## 快速开始

该 pipeline 可包装任意 drain。具体绑定方式取决于你的框架——请选择与你的框架匹配的标签页；下面每个示例都使用相同的形状。

::code-group
```typescript [Nuxt / Nitro]
// server/plugins/evlog-drain.ts
import type { DrainContext } from 'evlog'
import { createDrainPipeline } from 'evlog/pipeline'
import { createAxiomDrain } from 'evlog/axiom'

export default defineNitroPlugin((nitroApp) => {
  const pipeline = createDrainPipeline<DrainContext>()
  const drain = pipeline(createAxiomDrain())

  nitroApp.hooks.hook('evlog:drain', drain)
  nitroApp.hooks.hook('close', () => drain.flush())
})
```
```typescript [Next.js]
// lib/evlog.ts
import type { DrainContext } from 'evlog'
import { createEvlog } from 'evlog/next'
import { createDrainPipeline } from 'evlog/pipeline'
import { createAxiomDrain } from 'evlog/axiom'

const pipeline = createDrainPipeline<DrainContext>()
const drain = pipeline(createAxiomDrain())

export const { withEvlog, useLogger, log, createError } = createEvlog({
  service: 'my-app',
  drain,
})

// 在关闭前刷新（例如从你的自定义服务器或清理钩子中）
export const flushEvlog = () => drain.flush()
```
```typescript [Hono / Express / Fastify / Elysia / NestJS]
// 相同模式 — 将 `drain` 传递给框架的 evlog 中间件/模块
import type { DrainContext } from 'evlog'
import { createDrainPipeline } from 'evlog/pipeline'
import { createAxiomDrain } from 'evlog/axiom'

const pipeline = createDrainPipeline<DrainContext>()
const drain = pipeline(createAxiomDrain())

app.use(evlog({ drain })) // Hono / Express / Elysia
// await app.register(evlog, { drain }) // Fastify
// EvlogModule.forRoot({ drain }) // NestJS

// 在关闭时刷新
process.on('SIGTERM', () => drain.flush())
```
```typescript [Standalone]
// index.ts — 普通 TypeScript / Bun / Node 脚本
import type { DrainContext } from 'evlog'
import { initLogger } from 'evlog'
import { createDrainPipeline } from 'evlog/pipeline'
import { createAxiomDrain } from 'evlog/axiom'

const pipeline = createDrainPipeline<DrainContext>()
const drain = pipeline(createAxiomDrain())

initLogger({ drain })

// 在退出前刷新
await drain.flush()
```
::

::callout{icon="i-lucide-alert-triangle" color="warning"}
在进程退出前始终刷新 pipeline（`drain.flush()`）。在 Nitro 中使用 `close` 钩子；在独立脚本中在 `process.exit` 之前调用；在 serverless 运行时中使用 `waitUntil(drain.flush())`。
::

## 工作原理

::drain-pipeline-batching
::

事件会在抵达 `evlog:drain` 时被缓冲。当达到 `batch.size` 或 `batch.intervalMs` 到期时（以先发生者为准），就会刷新一个批次。失败时，会使用配置的 backoff 重试同一个批次；一旦 `retry.maxAttempts` 耗尽，就会调用 `onDropped` 传入丢失的事件。缓冲区由 `maxBufferSize` 限制——一旦满了，就会丢弃最旧的事件以保持内存占用平稳。

## 配置

下面的选项适用于任何框架——以与 [快速开始](#快速开始) 中相同的方式绑定生成的 `drain`。

```typescript [pipeline-config.ts]
import type { DrainContext } from 'evlog'
import { createDrainPipeline } from 'evlog/pipeline'
import { createAxiomDrain } from 'evlog/axiom'

const pipeline = createDrainPipeline<DrainContext>({
  batch: {
    size: 50,          // 每 50 个事件刷新一次
    intervalMs: 5000,  // 或每 5 秒刷新一次，以先到者为准
  },
  retry: {
    maxAttempts: 3,
    backoff: 'exponential',
    initialDelayMs: 1000,
    maxDelayMs: 30000,
  },
  maxBufferSize: 1000,
  onDropped: (events, error) => {
    console.error(`[evlog] 丢弃了 ${events.length} 个事件:`, error?.message)
  },
})

export const drain = pipeline(createAxiomDrain())
// 然后将 `drain` 绑定到你的框架——见上面的快速开始。
```

### 选项参考

| 选项 | 默认值 | 描述 |
|--------|---------|-------------|
| `batch.size` | `50` | 每个批次的最大事件数 |
| `batch.intervalMs` | `5000` | 刷新部分批次前的最长时间（毫秒） |
| `retry.maxAttempts` | `3` | 包括初始尝试在内的总尝试次数 |
| `retry.backoff` | `'exponential'` | `'exponential'` \| `'linear'` \| `'fixed'` |
| `retry.initialDelayMs` | `1000` | 首次重试的基础延迟 |
| `retry.maxDelayMs` | `30000` | 任意重试延迟的上限 |
| `maxBufferSize` | `1000` | 丢弃最旧事件前的最大缓冲事件数 |
| `onDropped` | - | 事件被丢弃时的回调（溢出或重试耗尽） |

## 退避策略

| 策略 | 延迟模式 | 使用场景 |
|----------|--------------|----------|
| `exponential` | 1秒、2秒、4秒、8秒... | 默认。最适合可能需要时间恢复的短暂失败 |
| `linear` | 1秒、2秒、3秒、4秒... | 可预测的延迟增长 |
| `fixed` | 1秒、1秒、1秒、1秒... | 每次延迟都相同。适用于限流 API |

## 返回的 Drain 函数

`pipeline(drain)` 返回的函数兼容 hook，并暴露以下内容：

| 属性 | 类型 | 描述 |
|----------|------|-------------|
| `drain(ctx)` | `(ctx: T) => void` | 将单个事件推入缓冲区 |
| `drain.flush()` | `() => Promise<void>` | 强制刷新所有缓冲事件 |
| `drain.pending` | `number` | 当前缓冲的事件数量 |

## 多个目标

使用单个 pipeline 包装多个适配器（一个批次会并行刷新到每个目标）：

```typescript [pipeline-fan-out.ts]
import type { DrainContext } from 'evlog'
import { createDrainPipeline } from 'evlog/pipeline'
import { createAxiomDrain } from 'evlog/axiom'
import { createOTLPDrain } from 'evlog/otlp'

const axiom = createAxiomDrain()
const otlp = createOTLPDrain()

const pipeline = createDrainPipeline<DrainContext>()
export const drain = pipeline(async (batch) => {
  await Promise.allSettled([axiom(batch), otlp(batch)])
})
// 按照快速开始中的方式精确地绑定 `drain`——Nitro 钩子、框架中间件，或 initLogger。
```

## 自定义 Drain 函数

你不需要适配器。传入任何接受批次的 async 函数：

```typescript [pipeline-custom.ts]
import type { DrainContext } from 'evlog'
import { createDrainPipeline } from 'evlog/pipeline'

const pipeline = createDrainPipeline<DrainContext>({ batch: { size: 100 } })

export const drain = pipeline(async (batch) => {
  await fetch('https://your-service.com/logs', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(batch.map(ctx => ctx.event)),
  })
})
// 将 `drain` 绑定到你的框架——见上面的快速开始。
```
