`evlog/toolkit` 提供了 `defineHttpDrain` —— 所有内置适配器都使用的同一个工厂。你只需提供两个纯函数（用于配置的 `resolve()`，以及用于载荷的 `encode()`），工具包会负责批处理、重试、超时和错误隔离。

在最底层，drain 仍然只是一个接收 `DrainContext` 并将数据发送到某处的函数——但实际上，你应该优先使用 `defineHttpDrain`。

构建一个自定义 evlog drain 适配器，将宽事件发送到我的目标端。

- 使用 evlog/toolkit 中的 defineHttpDrain——不要直接调用 fetch
- 提供返回配置（或返回 null 以跳过）的 resolve() 函数
- 提供返回 { url, headers, body }（或返回 null 以跳过）的 encode(events, config) 函数
- 配置请使用 evlog/toolkit 中的 resolveAdapterConfig，这样用户就能获得标准优先级（overrides → runtimeConfig → env）
- 使用标准化字段名：用于 bearer 密钥的 apiKey、基础 URL 的 endpoint、serviceName、timeout
- defineHttpDrain 会处理重试、超时、错误隔离和批处理——不要重新实现这些功能
- 通过 defineEvlog({ drain: createMyDrain() }) 或框架的 middleware drain 选项接入该 drain

文档：https://www.evlog.dev/adapters/building-blocks/custom
Pipeline：https://www.evlog.dev/adapters/building-blocks/pipeline

::

## 配方 — 15 行实现 `defineHttpDrain`

每个内置适配器都遵循的配方。将 `myservice` 替换为你的服务名。

```typescript [lib/my-drain.ts]
import {
  defineHttpDrain,
  resolveAdapterConfig,
  type ConfigField,
} from 'evlog/toolkit'

interface MyServiceConfig {
  apiKey: string
  endpoint?: string
  timeout?: number
}

const FIELDS: ConfigField<MyServiceConfig>[] = [
  { key: 'apiKey', env: ['MYSERVICE_API_KEY'] },
  { key: 'endpoint', env: ['MYSERVICE_ENDPOINT'] },
  { key: 'timeout' },
]

export function createMyServiceDrain(overrides?: Partial<MyServiceConfig>) {
  return defineHttpDrain<MyServiceConfig>({
    name: 'myservice',
    resolve: async () => {
      const cfg = await resolveAdapterConfig<MyServiceConfig>('myservice', FIELDS, overrides)
      if (!cfg.apiKey) {
        console.error('[evlog/myservice] 缺少 apiKey')
        return null
      }
      return cfg as MyServiceConfig
    },
    encode: (events, cfg) => ({
      url: `${cfg.endpoint ?? 'https://api.myservice.com'}/v1/ingest`,
      headers: {
        'Content-Type': 'application/json',
        Authorization: `Bearer ${cfg.apiKey}`,
      },
      body: JSON.stringify(events),
    }),
  })
}
```

就是这样。`defineHttpDrain` 会处理重试（默认 2 次）、超时（默认 5000ms）和错误隔离——即使你的目标端宕机，你的应用流水线也会继续运行。

然后将其接入你的框架：

::code-group

  nitroApp.hooks.hook('evlog:drain', drain)
})
```

export const { withEvlog, useLogger, log, createError } = createEvlog({

```

## 标准化配置优先级

`resolveAdapterConfig(namespace, fields, overrides)` 会按标准链路遍历，因此用户获得的配置体验与内置适配器保持一致：

1. 显式传给你工厂函数的 `overrides`
2. `runtimeConfig.evlog.<namespace>`（Nuxt/Nitro）
3. `runtimeConfig.<namespace>`（旧版 Nuxt/Nitro）
4. `NUXT_<NS>_<FIELD>` 环境变量
5. `<NS>_<FIELD>` 环境变量

字段名应遵循项目约定：`apiKey`、`endpoint`、`serviceName`、`timeout`。如果你在重命名已有字段（例如 `token` → `apiKey`），请在一个大版本周期内同时保留这两个 `ConfigField` 条目——参见 `axiom.ts` 和 `better-stack.ts` 中的弃用模式。

## 过滤和转换事件

`encode()` 会接收完整的 `WideEvent[]` 批次以及解析后的配置。可以在内部直接过滤或转换：

```typescript
encode: (events, cfg) => {
  const filtered = events.filter(e => e.level === 'error' && e.path !== '/health')
  if (filtered.length === 0) return null

  const payload = filtered.map(e => ({
    ts: new Date(e.timestamp).getTime(),
    severity: e.level.toUpperCase(),
    attributes: { method: e.method, path: e.path, status: e.status, duration: e.duration },
  }))

  return {
    url: `${cfg.endpoint}/v1/push`,
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(payload),
  }
}
```

从 `encode()` 返回 `null` 是一种干净的退出方式——该批次的 drain 将保持无操作。

## 当你无法使用 `defineHttpDrain` 时

如果你的目标端需要非 HTTP 传输（gRPC、websocket、厂商 SDK），可以下降一级使用 `defineDrain`：

```typescript
import { defineDrain, type DrainContext } from 'evlog/toolkit'

export const createCustomTransportDrain = () =>
  defineDrain<{ apiKey: string }>({
    name: 'custom',
    resolve: async () => ({ apiKey: process.env.MY_KEY! }),
    send: async (events, cfg) => {
      await myVendorSdk.publish(events, { token: cfg.apiKey })
    },
  })
```

你仍然可以获得配置解析、错误隔离和一致的结构——只是由你来负责底层传输。

## 批处理

对于高吞吐量场景，请使用 [Drain Pipeline](/adapters/building-blocks/pipeline) 来批量处理事件、在失败时重试，并自动处理缓冲区溢出：

```typescript [lib/my-drain.ts]
import type { DrainContext } from 'evlog'
import { createDrainPipeline } from 'evlog/pipeline'

const pipeline = createDrainPipeline<DrainContext>({
  batch: { size: 100, intervalMs: 5000 },
})

const drain = pipeline(async (batch) => {
  await fetch('https://api.example.com/logs/batch', {
    method: 'POST',
    body: JSON.stringify(batch.map(ctx => ctx.event)),
  })
})
```

## 错误处理 — 这些都已为你完成

`defineHttpDrain` 会自动强制执行每一项最佳实践：

1. **绝不抛出**——失败会被捕获并以 `[evlog/<name>]` 前缀记录。
2. **重试**——默认对瞬时错误尝试 2 次（可通过 `retries` 配置）。
3. **超时**——默认 5000ms（可通过 `timeout` 配置）。
4. **优雅降级**——`resolve()` 返回 `null` 会让 drain 变为无操作。

如果你因非 HTTP 传输而回退使用 `defineDrain`，请手动遵循相同规则——用 `try/catch` 包裹传输逻辑，使用 `console.error('[evlog/<name>] …')` 记录日志，并且绝不重新抛出。
