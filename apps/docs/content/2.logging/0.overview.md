evlog 提供了三种日志 API，每种都针对不同的使用场景设计。你可以在同一个项目中使用全部三种。

::callout{icon="i-lucide-info" color="info"}
这三种模式都共存于同一个日志记录器中。按调用场景选择即可——没有升级路径，没有高级模式，也没有需要切换的开关。底层使用相同的 drain、相同的脱敏机制、相同的类型。
::

::callout{icon="i-lucide-globe" color="neutral"}
不运行 HTTP 框架？请参见 [Standalone TypeScript](/frameworks/standalone) 和 [Cloudflare Workers](/frameworks/cloudflare-workers)。
::

## 三种模式

::card-group
  :::card
  ---
  icon: i-lucide-terminal
  title: 简单日志
  to: /logging/simple-logging
  color: neutral
  ---
  一个功能完整的通用日志记录器。用 `log.info`、`log.error`、`log.warn`、`log.debug` 替代 `console.log`、consola、pino 或 winston——具有相同的级别过滤、drain 管道、脱敏以及美化/JSON 输出。
  :::

  :::card
  ---
  icon: i-lucide-layers
  title: 大型事件
  to: /logging/wide-events
  color: neutral
  ---
  在一个工作单元（脚本、任务、队列任务或请求）中累积上下文，最终发出一个全面的事件。
  :::

  :::card
  ---
  icon: i-lucide-git-branch
  title: 请求日志
  to: /frameworks/overview
  color: neutral
  ---
  自动管理的、与 HTTP 请求绑定的大型事件。框架中间件会自动创建日志记录器并为你发出事件。
  :::
::

## 快速对比

### 简单日志（`log`）

每次调用一个事件。不累积上下文，也不管理生命周期。

```typescript [src/index.ts]
import { log } from 'evlog'

log.info('auth', '用户已登录')
log.error({ action: 'payment', error: 'card_declined', userId: 42 })
```

### 大型事件（`createLogger` / `createRequestLogger`）

每个工作单元一个事件。逐步累积上下文，完成时发出。

::code-group
```typescript [scripts/sync-job.ts]
import { createLogger } from 'evlog'

const log = createLogger({ jobId: 'sync-001', queue: 'emails' })
log.set({ batch: { size: 50, processed: 50 } })
log.emit()
```
```typescript [src/worker.ts]
import { createRequestLogger } from 'evlog'

const log = createRequestLogger({ method: 'POST', path: '/api/checkout' })
log.set({ user: { id: 1, plan: 'pro' } })
log.emit()
```
::

`createRequestLogger` 是 `createLogger` 的轻量封装，预填充了 `method`、`path` 和 `requestId`。

### 请求日志（框架中间件）

框架集成会在每个请求上自动创建一个大型事件日志记录器。`useLogger(event)` 用于获取已附加到请求上下文的日志记录器：

```typescript [server/api/checkout.post.ts]
import { useLogger } from 'evlog'

export default defineEventHandler(async (event) => {
  const log = useLogger(event)
  log.set({ user: { id: 1, plan: 'pro' } })
  return { success: true }
  // 在响应结束时自动发出
})
```

::callout{icon="i-lucide-info" color="info"}
`useLogger(event)` 并不会创建日志记录器，而是获取框架中间件已附加到事件上的日志记录器。每个框架有各自的获取方式（`useLogger`、`req.log`、`c.get('log')` 等）。在 Nuxt 中，`useLogger` 是自动导入的。
::

## 何时使用什么

| | `log` | `createLogger` / `createRequestLogger` | 框架中间件 |
|--|-------|----------------------------------------|---------------------|
| **使用场景** | 快速的一次性事件 | 脚本、任务、工作者、队列、无框架的 HTTP 请求 | 使用框架集成的 API 路由 |
| **上下文** | 单次调用 | 使用 `set()` 累积 | 使用 `set()` 累积 |
| **发射** | 立即发射 | 手动调用 `emit()` | 响应结束时自动发射 |
| **生命周期** | 无 | 手动管理 | 框架管理 |
| **输出** | 控制台 + 导出 | 控制台 + 导出 | 控制台 + 导出 + 增强 |

### 按上下文

| Context | Best fit | Why |
|---------|----------|-----|
| **Nuxt / Next / Hono / Express / … 中的 HTTP 路由** | 通过 [框架集成](/frameworks/overview) 使用 `useLogger(event)` | 每个请求一个大型事件，在响应结束时自动发出 |
| **没有框架的 HTTP 处理器** | `createRequestLogger({ method, path })` | 与框架中间件具有相同形状，手动发射 |
| **CLI 工具 / 一次性脚本** | 步骤用 `log.*`，运行摘要用 `createLogger` —— 参见 [Standalone](/frameworks/standalone) | 开发环境下更美观，CI 中结构化，为整个运行生成一个摘要事件 |
| **已发布的库** | 仅使用 `createLogger` —— 不要使用 `initLogger` —— 参见 [Standalone](/frameworks/standalone) | 不要污染宿主应用的全局配置，也不要强制消费者执行 drain |
| **后台任务 / 队列工作器 / cron** | 每次调用使用一个 `createLogger({ jobId, queue })` —— 参见 [Standalone](/frameworks/standalone) | 每次任务运行一个大型事件，非常适合重试分析 |
| **Cloudflare Worker / 边缘函数** | `createWorkersLogger(req)` 或 `createRequestLogger` —— 参见 [Cloudflare Workers](/frameworks/cloudflare-workers) | 每个请求一个事件，不需要 `process` 全局变量 |
| **AWS Lambda** | 先 `initLogger` 一次，再为每次调用创建一个 `createLogger` —— 参见 [AWS Lambda](/frameworks/aws-lambda) | 冷启动初始化、按事件作用域划分、在处理器中刷新 drain |
| **批处理 / 流水线步骤** | 每个阶段使用一个 `createLogger({ step })` | 每个阶段一个事件，输入和输出并排展示 |
| **AI 代理 / LLM 调用** | `createLogger` + [`createAILogger`](/logging/ai-sdk/overview) | 在同一个大型事件中记录 token 使用量、工具调用和流式指标 |
| **在请求内调用的库函数** | 从调用方使用 `useLogger(event)`，或将 logger 作为参数接收 | 继承父级请求上下文，贡献到同一个大型事件中 |
| **共享工作区包** | 将其视为库——参见 [Standalone](/frameworks/standalone) | 宿主应用负责 `initLogger` / drain；包使用 `createLogger` 或接受一个 logger |

::callout{icon="i-lucide-lightbulb" color="info"}
这些都不是另一种的“升级版”。在合适的地方，可以在同一个文件中同时使用 `log` 和 `createLogger`——它们共享全局 drain、脱敏和类型。
::

## 共享特性

所有三种模式都基于同一底层能力：

- **开发环境下使用美化输出**，**生产环境下使用 JSON**（默认，无需配置）
- **Drain 管道** 可将事件发送到 Axiom、Sentry、PostHog 等
- **结构化错误**，包含 `why`、`fix` 和 `link`，以及可选的仅后端可见 **`internal`** 日志
- **采样**，用于控制生产环境中的日志量
- **零依赖**，约 6 kB gzip
