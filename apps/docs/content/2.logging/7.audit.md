evlog 的审计层**不是一个并行系统**。审计事件是带有保留 `audit` 字段的宽事件。所有现有原语——drains、enrichers、redact、tail-sampling——都照常适用。只需添加 **1 个 enricher + 1 个 drain wrapper + 1 个 helper** 即可启用审计日志。

## 为什么需要审计日志？

合规框架（SOC2、HIPAA、GDPR、PCI）要求知道**谁在什么资源上、何时、从哪里、以什么结果做了什么**。evlog 无需第二套日志库即可满足这一点。

::tip
**审计事件是关于意图的事实，而不是对操作的测量。** 普通宽事件回答“这个请求表现如何？”（延迟、状态、token）。审计事件回答“谁试图做什么，是否被允许了？”。同一条流水线，不同的问题——这就是为什么 schema 是保留的，并且事件会在采样后被强制保留。
::

## 快速开始

你已经在使用 evlog 了。只需三处改动即可添加审计日志：

```typescript [server/plugins/evlog.ts]
import { auditEnricher, auditOnly, signed } from 'evlog'
import { createAxiomDrain } from 'evlog/axiom'
import { createFsDrain } from 'evlog/fs'

export default defineNitroPlugin((nitro) => {
  nitro.hooks.hook('evlog:enrich', auditEnricher())
  nitro.hooks.hook('evlog:drain', createAxiomDrain())
  nitro.hooks.hook('evlog:drain', auditOnly(
    signed(createFsDrain({ dir: '.audit' }), { strategy: 'hash-chain' }),
    { await: true },
  ))
})
```

::

就是这样。审计事件：

- 通过与你日志其余部分相同的宽事件流水线传递。
- 在 tail sampling 之后**始终保留**。
- 同时发送到你的主 drain（Axiom）以及一个专用、签名、仅追加的 sink（FS journal）。
- 通过 `auditEnricher` 自动携带 `requestId`、`traceId`、`ip` 和 `userAgent`。

::tip
**为什么需要两个 drains？** 主 drain（Axiom、Datadog、...）让审计数据与其余遥测数据保持在一起，因此仪表盘和查询仍然可用。签名 sink 是你的保险：如果主 drain 出故障、被清空，或者管理员悄悄删除了一行，FS journal 仍然保留这条链。审计人员两者都需要——既要快速查询，也要防篡改证据。
::

## 审计 Schema

`event.audit` 是每个宽事件上的一个类型化字段。下游查询会按 `audit IS NOT NULL` 进行过滤。

```typescript
interface AuditFields {
  action: string                          // 'invoice.refund'
  actor: {
    type: 'user' | 'system' | 'api' | 'agent'
    id: string
    displayName?: string
    email?: string
    // 对于 type === 'agent'，与 evlog/ai 字段对应：
    model?: string
    tools?: string[]
    reason?: string
    promptId?: string
  }
  target?: { type: string, id: string, [k: string]: unknown }
  outcome: 'success' | 'failure' | 'denied'
  reason?: string
  changes?: { before?: unknown, after?: unknown }
  causationId?: string                    // 导致此操作的动作 ID
  correlationId?: string                  // 一个操作中的所有动作共享
  version?: number                        // 默认为 1
  idempotencyKey?: string                 // 自动派生；适用于跨 drain 的安全重试
  context?: {                             // 由 auditEnricher 填充
    requestId?: string
    traceId?: string
    ip?: string
    userAgent?: string
    tenantId?: string
  }
  signature?: string                      // 由 signed({ strategy: 'hmac' }) 设置
  prevHash?: string                       // 由 signed({ strategy: 'hash-chain' }) 设置
  hash?: string
}
```

::tip
**`action` 的命名约定。** 使用 `noun.verb`（`invoice.refund`、`user.invite`、`apiKey.revoke`）。如果动作已经发生，用过去式（`invoice.refunded`）；如果 `withAudit()` 会解析结果，则用现在时。请在一个文件里维护一份小而固定的字典——审计人员和 SIEM 规则会按 `action` 查询，所以一个拼写错误就等于一个丢失的告警。
::

::warning
**不要伪造 actor。** 对 cron 任务、队列 worker 和后台任务使用 `actor.type: 'system'`；对由 token 认证的机器对机器调用使用 `actor.type: 'api'`；对 AI 工具调用使用 `actor.type: 'agent'`。把系统操作记成伪造的 `'user'` 是最容易在审计复核中失败的方式。
::

## 组合

每一层都是**可选且可替换**的。从视觉上看，审计事件在你的流水线中的路径如下：

```text
  log.audit / audit / withAudit
              │
              ▼
       set event.audit
              │
              ▼
    force-keep tail-sample
              │
              ▼
        auditEnricher
              │
              ▼
   redact + auditRedactPreset
              │
   ┌──────────┴──────────┐
   ▼                     ▼
 main drain         auditOnly(
 (Axiom /            signed(
  Datadog /          fsDrain))
  ...)
```

除 `log.audit`、`auditEnricher` 和 `auditOnly`/`signed` 之外的每个节点都与普通宽事件共享。

### Helper

`log.audit()` 是 `log.set({ audit: ... })` 加上 tail-sample 强制保留的语法糖：

```typescript
log.audit({
  action: 'invoice.refund',
  actor: { type: 'user', id: user.id },
  target: { type: 'invoice', id: 'inv_889' },
  outcome: 'success',
})

// 与以下内容完全等价：
log.set({ audit: { action: 'invoice.refund', /* ... */, version: 1 } })
```

`log.audit.deny(reason, fields)` 用于记录 AuthZ 被拒绝的动作——大多数团队都会忘记记录拒绝事件，但这恰恰是审计人员和安全团队最关心的：

::code-group

```typescript [Input]
if (!user.canRefund(invoice)) {
  log.audit.deny('权限不足', {
    action: 'invoice.refund',
    actor: { type: 'user', id: user.id },
    target: { type: 'invoice', id: invoice.id },
  })
  throw createError({ status: 403, message: 'Forbidden' })
}
```

### Enricher

`auditEnricher()` 会填充 `event.audit.context.{requestId, traceId, ip, userAgent, tenantId}`。如果你的策略不同，可以跳过它并提供自定义 enricher。

```typescript
nitro.hooks.hook('evlog:enrich', auditEnricher({
  tenantId: ctx => ctx.event.tenant as string | undefined,
  bridge: { getSession: async ctx => readSessionActor(ctx.headers) },
}))
```

### Drain Wrappers

::tip
**为什么要把审计过滤到单独的 sink？** 有三个原因：**成本**（审计量相较产品遥测很小——把它们分开存储，避免保留成本失控）、**权限**（审计数据集应该对工程师只读、对应用只写）、以及**保留期**（审计通常要保存 7 年以上；产品日志通常不会超过 90 天）。
::

`auditOnly(drain)` 只转发带有 `audit` 字段的事件。可与**任意** drain 组合：

```typescript
import { auditOnly } from 'evlog'
import { createAxiomDrain } from 'evlog/axiom'

// 将审计发送到专用的 Axiom 数据集：
nitro.hooks.hook('evlog:drain', auditOnly(
  createAxiomDrain({ dataset: 'audit', token: process.env.AXIOM_AUDIT_TOKEN }),
))
```

设置 `await: true` 使审计写入变为同步（审计不使用 fire-and-forget——默认具备崩溃安全）：

```typescript
auditOnly(createFsDrain({ dir: '.audit' }), { await: true })
```

`signed(drain, opts)` 会增加防篡改完整性。策略包括：

- `'hmac'` — 添加 `event.audit.signature`（规范化事件的 HMAC）。
- `'hash-chain'` — 添加 `event.audit.prevHash` 和 `event.audit.hash`，使序列形成可验证的链。若需要跨进程 / 持久链，请提供 `state: { load, save }`（Redis、文件、Postgres）。

::tip
**`signed()` 实际带来的价值。** 是检测，而不是预防。任何对底层 sink 有写权限的人仍然可以删掉文件或表——但这条链能事后证明哪些事件被删除或修改了。如果你已经写入只追加 / WORM 存储（S3 Object Lock、具有行级不可变性的 Postgres、仅追加的 BigQuery 表），就可以跳过 `signed()`；重复叠加完整性层只会增加延迟，却不会提高门槛。
::

```typescript
import { signed } from 'evlog'

signed(drain, { strategy: 'hmac', secret: process.env.AUDIT_SECRET! })

signed(drain, {
  strategy: 'hash-chain',
  state: {
    load: () => fs.readFile('.audit/head', 'utf8').catch(() => null),
    save: (h) => fs.writeFile('.audit/head', h),
  },
})
```

### Schema 纪律

在一个地方定义审计动作，避免魔法字符串：

```typescript
import { defineAuditAction } from 'evlog'

const refund = defineAuditAction('invoice.refund', { target: 'invoice' })

log.audit(refund({
  actor: { type: 'user', id: user.id },
  target: { id: 'inv_889' }, // type 被推断为 'invoice'
  outcome: 'success',
}))
```

::warning
**不要把整行数据库记录喂给 `auditDiff()`。** 在 diff 之前移除计算列、哈希密码、内部标志和大的 JSON blob。`changes` 的重点是*语义上发生了什么变化*（状态从 `paid` 变成了 `refunded`），而不是*字节层面发生了什么变化*（`lastModified` 时间戳跳了一下）。一个嘈杂的 `changes` 字段是让审计日志变得不可读的最快方式。
::

对于会修改数据的操作，请使用 `auditDiff()` 生成紧凑、感知脱敏的 JSON Patch：

::code-group

```typescript [Input]
import { auditDiff } from 'evlog'

const before = await db.users.byId(id)
const after = await db.users.update(id, patch)

log.audit({
  action: 'user.update',
  actor: { type: 'user', id: actorId },
  target: { type: 'user', id },
  outcome: 'success',
  changes: auditDiff(before, after, { redactPaths: ['password', 'token'] }),
})
```

```json [Output — changes patch]
{
  "audit": {
    "action": "user.update",
    "actor": { "type": "user", "id": "usr_42" },
    "target": { "type": "user", "id": "usr_99" },
    "outcome": "success",
    "changes": [
      { "op": "replace", "path": "/email", "from": "old@example.com", "to": "new@example.com" },
      { "op": "replace", "path": "/role", "from": "member", "to": "admin" },
      { "op": "replace", "path": "/password", "from": "[REDACTED]", "to": "[REDACTED]" }
    ],
    "version": 1,
    "idempotencyKey": "ak_5e7d8f9a0b1c2d3e"
  }
}
```

### 使用 `withAudit()` 自动埋点

开发者总会忘记调用 `log.audit()`。把函数包起来，就不会漏记：

::tip
**何时包装，何时手动调用。** 为那些*纯粹属于审计价值的动作*（退款、删除、角色变更、重置密码）使用包装——结果解析会自动完成，你也不会意外漏掉调用。若审计只是更大处理程序中的多个决策之一，或者你需要在动作完成前发出审计（例如“用户请求删除”），则保留手动 `log.audit()`。
::

::code-group

```typescript [Input]
import { withAudit, AuditDeniedError } from 'evlog'

const refundInvoice = withAudit(
  { action: 'invoice.refund', target: input => ({ type: 'invoice', id: input.id }) },
  async (input: { id: string }, ctx) => {
    if (!ctx.actor) throw new AuditDeniedError('匿名退款被拒绝')
    return await db.invoices.refund(input.id)
  },
)

await refundInvoice({ id: 'inv_889' }, {
  actor: { type: 'user', id: user.id },
  correlationId: requestId,
})
```

```json [Output — success]
{
  "audit": {
    "action": "invoice.refund",
    "actor": { "type": "user", "id": "usr_42" },
    "target": { "type": "invoice", "id": "inv_889" },
    "outcome": "success",
    "version": 1,
    "idempotencyKey": "ak_8f3c4b2a1e5d6f7c",
    "correlationId": "a566ef91-7765-4f59-b6f0-b9f40ce71599"
  }
}
```

```json [Output — failure]

    "name": "StripeError",
```

结果解析：

- `fn` 解析成功 → `outcome: 'success'`。

- 其他抛出的错误 → `outcome: 'failure'`，然后重新抛出。

## 合规

### 完整性

对审计日志进行哈希链处理，以便任何篡改都能被检测出来。每个事件的哈希都包含前一个哈希，因此删除某一行会使该点之后的链断裂。

```typescript
auditOnly(
  signed(createFsDrain({ dir: '.audit' }), { strategy: 'hash-chain' }),
  { await: true },
)
```

::note
用于遍历并验证链的 CLI（`evlog audit verify`）已列入规划。与此同时，可通过重新计算已存储事件的哈希，并将每个 `prevHash` 与前一个事件的 `hash` 比较来进行验证。
::

::warning
**请每年轮换用于 HMAC 签名审计的 `secret`。** 轮换时，请在签名旁边嵌入一个密钥 ID（例如通过 `declare module` 为 `AuditFields` 扩展 `keyId`），这样旧事件仍可使用之前的 secret 验证。验证器应按 ID 查找密钥，而不是假设只有一个全局 secret。
::

### 脱敏

审计事件会经过你现有的 `RedactConfig`。可与严格审计预设组合，以强化对 PII 的处理：

```typescript
import { auditRedactPreset } from 'evlog'

initLogger({
  redact: {
    paths: [
      ...(auditRedactPreset.paths ?? []),
      'user.password',
    ],
  },
})
```

该预设会移除 `Authorization` / `Cookie` 头，以及常见凭证字段名（`password`、`token`、`apiKey`、`cardNumber`、`cvv`、`ssn`）——无论它们出现在 `audit.changes.before` 还是 `audit.changes.after` 中。

### GDPR 与仅追加式存储

仅追加式审计日志与 GDPR 的“被遗忘权”存在冲突。当前推荐的模式是：

1. 保持审计行不可变。
2. 使用按参与者分配的密钥加密 PII 字段（该密钥保存在审计存储之外）。
3. 若要“忘记”某个用户，只需删除其密钥——审计行仍保留，链仍然有效，而 PII 变得不可读。

内置的 `cryptoShredding` 辅助工具已列入 [后续规划](https://github.com/HugoRCD/evlog/issues)。

### 保留策略

保留策略按设计属于存储层职责。evlog 的审计层不会强制保留窗口，因为所有受支持的 sink 已经各自提供了更强且可审计的机制。请选择与你的 sink 匹配的方式：

- **FS** — 将 `createFsDrain({ maxFiles })` 与每日压缩器结合使用。
- **Postgres** — 计划执行 `DELETE FROM audit_events WHERE timestamp < now() - interval '7 years'`。
- **Axiom / Datadog / Loki** — 在平台中设置数据集保留策略。

请在安全策略中记录所选窗口。审计员关心的是书面规则，而不是执行该规则的组件。

### 常见陷阱

- **只记录成功。** 审计员最关注的是拒绝。务必在每个授权检查的否定分支同时调用 `log.audit()` 与 `log.audit.deny()`。
- **通过 `changes` 泄露 PII。** `auditDiff()` 会经过你的 `RedactConfig`，但前提是字段路径已被列出。把 `password`、`token`、`apiKey` 等一次性全局添加进去，这样以后就无需再为此操心。
- **把审计当作可观测性。** 不要对审计事件进行采样、降采样或汇总。默认开启强制保留——不要关闭它。
- **将 `actor.id` 与会话 ID 混淆。** `actor.id` 是稳定的用户 ID（或系统身份）。应通过 `context.requestId` / `context.traceId` 关联会话，绝不要通过 actor 来关联。
- **忘记独立任务。** 定时任务、队列 worker 和 CLI 也会触发需要审计的操作。使用 `audit()`（无请求）或 `withAudit()` 来保持与 HTTP 路由一致的覆盖率。

## 实战方案

### 将审计日志写入磁盘

::code-group

```typescript [Input — server/plugins/evlog.ts]
import { auditOnly, signed } from 'evlog'
import { createFsDrain } from 'evlog/fs'

nitro.hooks.hook('evlog:drain', auditOnly(
  signed(createFsDrain({ dir: '.audit', maxFiles: 30 }), { strategy: 'hash-chain' }),
  { await: true },
))
```

```ndjson [Output — .audit/2026-04-24.ndjson]
{"audit":{"action":"invoice.refund","actor":{"type":"user","id":"usr_42"},"target":{"type":"invoice","id":"inv_889"},"outcome":"success","version":1,"idempotencyKey":"ak_8f3c4b2a1e5d6f7c","prevHash":null,"hash":"3f2c8e1a..."}}
{"audit":{"action":"user.update","actor":{"type":"user","id":"usr_42"},"target":{"type":"user","id":"usr_99"},"outcome":"success","version":1,"idempotencyKey":"ak_5e7d8f9a0b1c2d3e","prevHash":"3f2c8e1a...","hash":"9a1b4d7c..."}}
```

::

每一行的 `prevHash` 都与上一行的 `hash` 匹配。对任意一行的篡改都会使该点之后的链断裂——验证器会重新播放哈希并报告第一个不匹配处。

### 将审计日志写入专用的 Axiom 数据集

::code-group

```typescript [Input — server/plugins/evlog.ts]
import { auditOnly } from 'evlog'
import { createAxiomDrain } from 'evlog/axiom'

nitro.hooks.hook('evlog:drain', createAxiomDrain({ dataset: 'logs' }))
nitro.hooks.hook('evlog:drain', auditOnly(
  createAxiomDrain({ dataset: 'audit', token: process.env.AXIOM_AUDIT_TOKEN }),
))
```

```kusto [Output — Axiom query]
['audit']
| where audit.action == "invoice.refund"
| summarize count() by audit.outcome, bin(_time, 1h)
```

```kusto [Output — denials by actor]
['audit']
| where audit.outcome == "denied"
| summarize count() by audit.actor.id, audit.action
| order by count_ desc
```

::

将数据集拆分后，审计数据集可以拥有更长的保留期（7 年）、更严格的访问控制，以及单独的计费项——而无需触碰你管道的其他部分。

### 将审计日志写入 Postgres

::code-group

```typescript [Input — server/plugins/evlog.ts]
import { auditOnly } from 'evlog'
import type { DrainContext } from 'evlog'

const postgresAudit = async (ctx: DrainContext) => {
  await db.insert(auditEvents).values({
    id: ctx.event.audit!.idempotencyKey,
    timestamp: new Date(ctx.event.timestamp),
    payload: ctx.event,
  }).onConflictDoNothing()
}

nitro.hooks.hook('evlog:drain', auditOnly(postgresAudit, { await: true }))
```

```sql [Output — audit_events row]
SELECT id, timestamp, payload->'audit'->>'action' AS action,
       payload->'audit'->>'outcome' AS outcome
FROM audit_events
WHERE id = 'ak_8f3c4b2a1e5d6f7c';

--          id          |       timestamp       |     action      | outcome
-- ---------------------+-----------------------+-----------------+---------
--  ak_8f3c4b2a1e5d6f7c | 2026-04-24 10:23:45.6 | invoice.refund  | success
```

::

确定性的 `idempotencyKey` 使重试变得安全——重复插入会通过 `ON CONFLICT DO NOTHING` 合并掉。若没有它，重试期间一次短暂的网络波动就会创建重复的审计行，而这恰恰是你不想要的。

## 测试

`mockAudit()` 会捕获测试期间发出的每一条审计事件：

```typescript
import { mockAudit } from 'evlog'

it('refunds the invoice and records an audit', async () => {
  const captured = mockAudit()

  await refundInvoice({ id: 'inv_889' }, { actor: { type: 'user', id: 'u1' } })

  expect(captured.events).toHaveLength(1)
  expect(captured.toIncludeAuditOf({
    action: 'invoice.refund',
    target: { type: 'invoice', id: 'inv_889' },
    outcome: 'success',
  })).toBe(true)

  captured.restore()
})
```

## API 参考

| 符号 | 类型 | 说明 |
|---|---|---|
| `AuditFields` | type | wide event 上保留字段 |
| `defineAuditAction(name, opts?)` | factory | 带类型的动作注册表，推断 target 形状 |
| `log.audit(fields)` | method | `log.set({ audit })` + 强制保留 的语法糖 |
| `log.audit.deny(reason, fields)` | method | 记录被拒绝的操作 |
| `audit(fields)` | function | 供脚本 / 作业独立使用 |
| `withAudit({ action, target })(fn)` | wrapper | 自动发出成功 / 失败 / 拒绝 |
| `auditDiff(before, after)` | helper | 感知脱敏的 `changes` JSON Patch |
| `mockAudit()` | test util | 在测试中捕获 + 断言审计 |
| `auditEnricher(opts?)` | enricher | 自动填充请求 / 运行时 / 租户上下文 |
| `auditOnly(drain, { await? })` | wrapper | 仅路由带有 `audit` 字段的事件 |
| `signed(drain, opts)` | wrapper | 通用完整性包装器（hmac / hash-chain） |
| `auditRedactPreset` | config | 审计事件的严格 PII 处理 |

所有内容都从主 `evlog` 入口导出。
